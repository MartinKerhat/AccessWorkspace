# Object Model Specification

## Purpose

This document defines how the four main object categories should be modeled:

- `Connections`
- `Key Vault`
- `App registrations`
- `Passwords`

For each category it defines:

- required attributes
- optional attributes
- source of truth
- what is stored locally
- what must not be duplicated locally
- retrieval behavior

The goal is to make the next backend and admin/detail refactor deliberate instead of field-by-field improvisation.

## Core design rules

### 1. One workspace, category-specific objects

The product is one workspace, but each category has its own structure and workflow. We should not force all categories into one generic field set beyond a small shared base.

### 2. Metadata may be cached, secrets should not be duplicated without reason

The local database can store operational metadata for search, ownership, authorization, UI rendering, and audit.

It should not become a second secret store for external systems.

### 3. System of record matters

Some objects are authored inside this app.
Some are authored elsewhere and only surfaced here.

This distinction should be explicit for every category.

### 4. Separate operational metadata from sensitive values

The app may store:

- labels
- owner mappings
- allowed groups
- notes
- launch settings
- imported metadata

The app should avoid storing external secret values when they already belong to Azure Key Vault or another source of truth.

### 5. Retrieval should reflect object origin

- manual objects are read from the local database
- external objects are retrieved from integrations and optionally cached locally as metadata snapshots
- sensitive secret values are fetched on demand

## Shared base fields

These fields are useful across all object categories.

### Required base fields

- `id`
- `category`
- `name`
- `description`
- `owner`
- `allowed_groups`
- `status`
- `created_at`
- `updated_at`

### Optional base fields

- `owner_team`
- `environment`
- `archived_at`
- `source_system`
- `source_object_id`
- `last_synced_at`
- `notes`

### Capability flags

Not every category needs every capability, but these should be modeled explicitly:

- `view_allowed`
- `launch_allowed`
- `reveal_allowed`
- `copy_allowed`
- `edit_allowed`

## Category: Connections

## Purpose

Represents operational connection endpoints, mainly SSH and RDP.

These objects are intended to become the foundation for replacing Royal TS-like workflows with a backend-governed and later launcher-assisted native client path across Windows, macOS, and Linux.

## Source of truth

Primary source of truth:

- this app

Reason:

These are operational workspace objects that usually need app-specific metadata, ownership, groups, launch rules, and notes.

## Local storage strategy

Stored locally:

- all connection metadata
- launch configuration metadata
- folder path / grouping metadata
- allowed groups
- ownership
- notes
- optional secret reference
- encrypted app-managed password or passphrase values when the connection is not using an external provider reference
- optional per-user override reference that points to a Password object owned or selected by the connecting user

Should not be duplicated locally unless explicitly configured:

- secret values that already live in Key Vault

## Required attributes

- `name`
- `connection_type` (`ssh`, `rdp`)
- `target_host`
- `owner`
- `allowed_groups`
- `launch_allowed`

## Optional attributes

- `target_port`
- `username`
- `folder_path`
- `launch_mode`
- `domain`
- `window_mode`
- `screen_mode`
- `admin_session`
- `automatic_logon`
- `use_multiple_monitors`
- `show_connection_bar`
- `mac_address`
- `secret_ref`
- `description`
- `environment`
- `launch_profile`
- `working_notes`
- `reveal_allowed`
- `copy_allowed`
- `personal_password_override_ref` as a per-user relation, not as a field duplicated directly on the shared Connection row

## Retrieval behavior

- list/get from local database
- app-managed secret value decrypted only when reveal or launcher execution needs it
- external secret value retrieved only if needed via secret provider
- launch payload generated by backend
- launch credential selection resolved by backend as `personal override -> shared/default connection secret`
- later helper consumes launch payload locally

## Detail-view focus

- host
- port
- username
- launch capability
- owner
- allowed groups
- notes

## Category: Key Vault

## Purpose

Represents Azure Key Vault objects surfaced in a better operational UI.

Initial focus:

- secrets

Future room:

- other Key Vault object types if relevant later

## Source of truth

Primary source of truth:

- Azure Key Vault

Reason:

The app should not duplicate Key Vault secret values or become a parallel Key Vault store.

## Local storage strategy

Stored locally:

- object identity
- vault name
- object name
- object type
- URI/reference
- owner mappings
- allowed groups
- notes added by this app
- non-sensitive metadata snapshot
- last sync timestamp
- expiration metadata

Should not be duplicated locally:

- actual secret value from Key Vault
- version contents

## Required attributes

- `name`
- `vault_name`
- `object_type` initially `secret`
- `object_name`
- `source_object_id` or canonical URI
- `owner`
- `allowed_groups`

## Optional attributes

- `version`
- `content_type`
- `expires_at`
- `not_before`
- `enabled`
- `last_synced_at`
- `description`
- `notes`
- `reveal_allowed`
- `copy_allowed`

## Retrieval behavior

Metadata list/get:

- retrieved from Azure integration
- cached locally for search and UI speed

Secret value reveal:

- fetched on demand from Key Vault
- never read from local stored duplicate
- audited every time

## Key rule

The local DB may cache Key Vault metadata, but it should not persist the live secret payload as the normal operating model.

## Detail-view focus

- vault
- object name
- current version or selected version
- expiry
- content type
- owner
- allowed groups
- reveal/copy policy

## Category: App registrations

## Purpose

Represents application registrations and related credentials that matter operationally, especially around client secrets and expiration visibility.

## Source of truth

Primary source of truth:

- Azure / Entra / external provider system (Atlassian, Google...)

Reason:

Application registrations and their credential metadata already exist outside this app. The app should surface and govern visibility, not become the master record.

## Local storage strategy

Stored locally:

- application metadata snapshot
- ownership
- allowed groups
- app classification
- last sync metadata
- expiry metadata

Should not be duplicated locally:

- external client secret values
- external authoritative credential records when they already exist in Entra/provider

## Required attributes

- `name`
- `provider`
- `application_id`
- `owner`
- `allowed_groups`

## Optional attributes

- `tenant_id`
- `client_id`
- `credential_type`
- `credential_expires_at`
- `linked_secret_ref`
- `description`
- `notes`
- `last_synced_at`
- `status`
- `display_name_external`

## Retrieval behavior

Metadata list/get:

- retrieved from Azure/Entra integration or other provider integration
- cached locally for search, filtering, ownership overlays, and expiry views

Secret reveal:

- should not reveal as this value is usually not retrievable after creation

## Key rule

App registration expiry should be surfaced here, but not duplicated as local secret truth.

## Detail-view focus

- application id / client id
- provider
- owner
- credential type
- credential expiry
- linked Key Vault reference if present

## Category: Passwords

## Purpose

Represents shared login/password style entries for websites, tools, and legacy systems.

This is the category for the objects that are actually authored inside this app most often.

This category now also acts as the first reusable credential store for per-user Connection overrides.

## Source of truth

Primary source of truth:

- this app

Possible secret provider options:

- inline for development/demo only
- future external provider reference
- Key Vault reference if desired later

## Local storage strategy

Stored locally:

- operational metadata
- target system / URL
- username
- owner
- allowed groups
- notes
- secret reference metadata
- personal visibility flag
- creator identity for personal objects

Stored locally as value only when intentionally using app-managed password mode:

- password value

## Required attributes

- `name`
- `target_system`
- `owner`
- `allowed_groups`
- `username` if the object is credential-based
- `password`

## Optional attributes

- `target_url`
- `secret_ref`
- `password_mode` (`inline`, `external_reference`, `key_vault_reference`)
- `description`
- `notes`
- `reveal_allowed`
- `copy_allowed`
- `expires_at`
- `personal`

## Retrieval behavior

Metadata list/get:

- from local database

Secret reveal:

- inline if app-managed password
- external fetch if provider-backed later

## Key rule

Passwords are the category most likely to be truly app-authored and locally managed, unlike Key Vault and app registrations.

For Connection overrides, the Password object remains the only place where the reusable username/password pair lives. Override records should reference the Password object instead of copying the secret into another table.

## Detail-view focus

- target system
- target URL
- username
- owner
- reveal/copy actions
- notes

## Source-of-truth summary

### Authored in this app

- `Connections`
- `Passwords` mostly

### Retrieved from external source and surfaced here

- `Key Vault`
- `App registrations`

### Hybrid metadata model

Even for externally sourced objects, this app may still own:

- owner mapping
- allowed groups
- notes
- UI categorization

## Storage recommendation by category

| Category | Metadata stored locally | Secret value stored locally | External fetch on demand |
| --- | --- | --- | --- |
| Connections | Yes | Only if intentionally app-managed | Yes, for referenced secrets |
| Key Vault | Yes | No | Yes |
| App registrations | Yes | No | Yes, via linked provider/reference |
| Passwords | Yes | Sometimes, depending on mode | Yes, if provider-backed |

## Recommended implementation path

### Step 1

Introduce category-specific schemas in code while preserving a small shared base model.

### Step 2

Separate "manual objects" from "externally sourced objects" in persistence and service logic.

### Step 3

Introduce explicit source metadata:

- `source_kind` (`manual`, `azure_key_vault`, `entra_app_registration`, etc.)
- `source_object_id`
- `sync_state`
- `last_synced_at`

### Step 4

Refactor admin UI so:

- Connections and Passwords are created/edited locally
- Key Vault and App registrations are primarily linked/imported/synced rather than fully created manually

### Step 5

Add provider-backed retrieval rules:

- local metadata
- external sensitive fetch on demand

## What should not be duplicated

The app should not normally duplicate:

- Azure Key Vault secret values
- Azure/Entra app registration client secret values
- external authoritative expiry truth when it can be synced reliably

The app should duplicate only:

- searchable metadata
- operational overlays such as ownership, rights, and notes
- cached snapshots needed for UX and reporting
